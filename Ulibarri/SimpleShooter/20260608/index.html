162 Connecting Animation To Gameplay¶

1. 왜 게임플레이가 애니메이션을 주도해야 하는가?¶

  • 원칙: 애니메이션이 캐릭터 이동을 결정하는 것이 아니라, 캐릭터의 이동 속도에 맞춰 애니메이션이 따라와야 합니다.

  • 이유: 게임 디자이너가 "캐릭터가 너무 느린 것 같으니 속도를 키우자"라고 할 때, 애니메이션에 묶여 있으면 수정하기 어렵기 때문입니다. 게임의 '조작감(Game Feel)'을 먼저 자유롭게 수정할 수 있도록 둘을 분리해야 합니다.

2. 애님 그래프(AnimGraph) 설정하기¶

1. 블렌드 스페이스 배치¶

  • ABP_ShooterCharacter에 BS_Locomotion 배치, 노드 연결

2. BS_Locomotion을 제어할 Angle, Speed 변수 생성¶

3. 컴파일 후 Speed 값과 Angle 값을 detail 에서 수정해보고 반영되는지 확인¶

3. EventGraph에서 실제 속도 가져오기¶

1. 업데이트 이벤트 활용¶

  • Blueprint Update Animation 노드를 사용

2. 속도(Velocity) 추출¶

  • Get Pawn Owner -> Get Velocity 노드로 현재 이동벡터(방향과 속력 포함)

3. 속력(Speed) 계산¶

  • Get Velocity-> Vector Length 노드의 반환값, 단위시간당 이동속력(cm/s)을 받아와서 Speed에 연결

4. 변수 세팅 및 결과 확인¶

  • 정상 작동 : 가만히 있을 때 Idle 상태였다가 이동하면 속도에 맞추서 Walk/Jog 애니메이션 출력
  • 문제점
    • angle이 아직 연결되지 않아 대각선이나 옆으로 이동시 동작이상
    • 캐릭터 실제 이동 속도와 애니메이션 재생 속도 동기화가 안되어서 Foot Sliding 현상
      • BS_Locomotion의 Speed 경우 0~100 사이로 처리되는데 수정할 필요있음.

163 Inverse Transforming Vectors¶

글로벌(Global) vs 로컬(Local)¶

  • 왜 로컬 공간이 필요한가?: 캐릭터가 월드 기준으로 어느 방향으로 가든, 애니메이션은 "캐릭터 정면 기준으로 오른쪽으로 가나? 뒤로 가나?"를 알아야 그에 맞는 꽃게걸음(A, D 키)이나 뒷걸음질(S 키) 애니메이션을 틀어줄 수 있기 때문입니다.

위치 벡터 vs 방향 벡터의 변환시 차이점¶

  • 글로벌(월드)과 로컬(지역) 좌표계 사이를 왔다 갔다 변환(Transform)할 때
데이터 종류 변환 시 고려하는 요소 변환 후 좌표 변화 언리얼 노드 예시
위치 벡터
(Location)		 부모의 회전, 크기 + [현재 위치] 모두 고려 기준점이 바뀌면 좌표 전체가 통째로 이동함 `Inverse Transform Location
방향 벡터
(Direction / Velocity)		 부모의 회전, 크기만 고려 ([현재 위치]는 무시) 기준점이 어디든 화살표의 회전과 크기만 바뀜 `Inverse Transform Direction

블루프린트 구현 순서¶

1. 글로벌을 로컬로 역변환 (Inverse Transform)¶

  • 로벌 속도를 로컬로 바꾸어야 하므로 역변환(Inverse Transform) 노드 사용
    Try Get Pawn Owner +-> Get Velocity -------+ 
                   |                       |
                   +-> Get Actor Transform -> Inverse Transform direction

¶

2. 변환된 벡터에서 회전각(Yaw) 추출하기¶

  • 로컬로 바뀐 벡터(예: 캐릭터 정면 기준 대각선 방향 벡터 등)에서 순수한 수평 각도(Z축 회전인 Yaw)를 뽑아내야 합니다.
      -> Rotation from X Vector -> Return Value Z(Yaw) -> Set Speed

164 Calculating Animation Speeds¶

발 밀림(Foot Sliding) 현상이 발생하는 이유¶

  • 게임 속 캐릭터의 이동 속도(기획/코드상의 속도)와 재생되는 애니메이션의 속도가 서로 일치하지 않기 때문에 발생
  • 예시: 게임 안에서 캐릭터는 초당 100cm($100\text{ cm/s}$)로 천천히 이동하고 있는데, 재생되는 애니메이션은 초당 300cm로 달리는 모션이라면 발이 바닥에서 헛돌며 미끄러지는 것처럼 보이게 됩니다.
  • 해결책: 애니메이션 속 발이 실제로 이동하는 속도를 정확히 계산해서 엔진의 블렌드 스페이스 Speed축(Axis) 값에 입력해 주어야 합니다.

애니메이션 속도 계산 공식¶

  • 발이 바닥에 닿아 있는 동안, 발목 뼈(Ankle Bone)가 이동한 거리와 걸린 시간을 이용해 속도를 구합니다.
      속도(Speed) = (종료 위치 - 시작 위치) / (종료 시간 - 시작 시간)
  • 주의할 점
    • Unreal Engine 5에서는 뷰포트에서 뼈를 직접 클릭하기 어려우므로, 왼쪽 스켈레톤(Skeleton) 탭에서 검색창을 이용해 Foot_R (오른발 발목 뼈)을 선택해야 합니다. 또한, 로컬 좌표계 시스템에서는 위치 값이 변하지 않으므로 반드시 월드 좌표계(World Location) 로 전환한 뒤 값을 확인해야 합니다. 여기서는 전진 방향인 Y축 값을 기준으로 계산합니다.

달리기 & 걷기¶

계산할 때 기준을 잡는 팁¶

1. 뼈(Bone)의 수치 변화로 판단하기 (가장 정확함)¶
  • 공중에 떠 있을 때: 캐릭터의 전진 속도나 모션에 따라 Y축 값이 불규칙하게 변하거나 앞으로 나아갑니다.
  • 바닥에 디뎠을 때: 발이 바닥에 고정되면서, 캐릭터가 제자리에서 달리는 애니메이션 특성상 Y축 값이 일정한 방향(뒤쪽)으로 일정하게 줄어들기(또는 늘어나기) 시작합니다.
2. 일반적인 애니메이션 스타일별 기준¶
  • 일반적인 걷기 (Heel Strike): 뒤꿈치가 바닥에 닿는 순간부터 이미 발은 땅에 고정되어 뒤로 밀리기 시작합니다. 따라서 뒤꿈치가 닿는 첫 프레임을 선택하는 것이 맞습니다.
  • 조깅 / 런닝 (Mid-foot / Fore-foot Strike): 속도가 빠른 달리기 애니메이션은 뒤꿈치가 아니라 발 앞꿈치나 발바닥 전체가 동시에 쿵 하고 닿는 경우가 많습니다. 이때는 발바닥이 바닥에 완전히 밀착되는 첫 프레임을 잡으시면 됩니다.

조깅(달리기) 애니메이션 계산 (Jog Forward Combat)¶

  • 이동 거리: -28 - 17 = -45cm (뒤로 밀어낸 거리)
  • 걸린 시간: 0.30 - 0.16 = 0.14초
  • 속도 결과: -45 / 0.14 = 약 325cm/s

걷기 애니메이션 계산 (Walk Forward)¶

  • 이동 거리: 대략 -42cm
  • 걸린 시간: 0.67 - 0.34 = 0.33초
  • 속도 결과: -42 / 0.33 = 약130cm/s

블렌드 스페이스(Blend Space)에 적용하기¶

  • 계산된 소수점 자릿수의 정확한 값을 그대로 쓰면 그리드(Grid) 나눗셈이 복잡해지므로, 50 단위로 반올림(올림)하여 값을 단순화
    • 걷기 속도 (130cm/s) -> 150
    • 달리기 속도 (325cm/s) -> 350

블렌드 스페이스 축 설정 세팅¶

  • 최대 축 값 (Maximum Axis Value): 350
  • 그리드 분할 수 (Grid Divisions): 7 (350을 50 단위로 나누기 위함: 350 / 50 = 7)

BP_Character 설정 세팅¶

  • CharacterMovement Component의 MaxWalkSpeed도 달리기 최대 속도랑 맞춰준다.
