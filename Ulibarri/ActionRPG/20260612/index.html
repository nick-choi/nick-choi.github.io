The Actor Class¶

요약¶

  • 첫 클래스로 Actor 선택: 언리얼 월드의 모든 것은 Actor에서 파생되므로 첫 클래스로 가장 적합합니다. 블루프린트 프로젝트도 C++ 클래스를 추가하면 C++ 프로젝트로 전환됩니다.

  • Public/Private 구조 분리: 헤더 파일(.h)은 Public 폴더에, 소스 파일(.cpp)은 Private 폴더에 저장하여 프로젝트를 깔끔하게 관리합니다.

  • 기본 생성 코드(Boilerplate) 이해: 리플렉션 시스템을 위한 매크로(UCLASS(), GENERATED_BODY())와 기본 함수(BeginPlay(), Tick())의 역할을 이해합니다.

  • 코드 정리: 불필요한 기본 주석과 중복된 public 섹션을 정리하여 가독성을 높입니다.

상세 정리¶

1. C++ 클래스 생성 및 폴더 구조 설정¶

  • 생성 방법: 에디터 상단 메뉴에서 Tools -> New C++ Class 선택.
  • 부모 클래스 선택: 공통 클래스 중 Actor를 선택 (Actor는 레벨에 배치하거나 스폰할 수 있는 오브젝트의 최상위 기본 클래스임).
  • 경로 및 이름 설정: 클래스 이름을 Item으로 지정하고, Items라는 별도 폴더를 만들어 관리.
  • 클래스 타입(Public/Private):
    • Public을 선택하면 헤더 파일(Item.h)은 Public/Items/ 폴더에, 소스 파일(Item.cpp)은 Private/Items/ 폴더에 자동으로 나누어 저장됩니다. (실무 및 대형 프로젝트에서 자주 쓰이는 방식)

2. 콘텐츠 브라우저 및 솔루션 확인¶

  • 클래스를 생성하면 콘텐츠 드로어(Ctrl + Space)에 C++ Classes 폴더가 생기며, 그 안에 우리가 만든 Item 클래스가 나타납니다.
  • 프로젝트 이름(스크립트에서는 Lecture)의 모듈 구조를 보면 기본적으로 포함되는 빌드 파일(.Build.cs)과 프로젝트 기본 파일(Lecture.h, Lecture.cpp)이 자동 생성되어 있습니다.

3. 헤더 파일 (Item.h) 구조 분석¶

 once

 "CoreMinimal.h"
 "GameFramework/Actor.h"
 "Item.generated.h"

UCLASS()
class LECTURE_API AItem : public AActor
{
    GENERATED_BODY()

public: 
    AItem();
    virtual void Tick(float DeltaTime) override;

protected:
    virtual void BeginPlay() override;
};
  • once: 헤더 파일이 중복으로 include되는 것을 방지합니다.
  • "GameFramework/Actor.h": Actor 클래스를 상속받기 위해 필수적인 헤더입니다.
  • LECTURE_API: 이 클래스를 DLL(동적 링크 라이브러리)에서 사용할 수 있도록 언리얼이 자동으로 붙여주는 매크로입니다.
  • UCLASS() 및 GENERATED_BODY(): 언리얼 엔진의 핵심 기능인 리플렉션(Reflection) 시스템에 클래스를 등록하는 매크로입니다. 이를 통해 블루프린트와 연동되거나 가비지 컬렉션 등의 혜택을 받게 됩니다.

4. 핵심 기본 함수 (주요 라이프사이클)¶

  • 생성자 (AItem()): 오브젝트가 생성될 때 프로퍼티의 기본값을 설정합니다.
  • BeginPlay(): 게임이 시작되거나 이 Actor가 월드에 스폰될 때 딱 한 번 호출되는 가상(Virtual) 함수입니다.
  • Tick(float DeltaTime): 게임의 매 프레임(화면이 업데이트될 때마다)마다 매우 빠르게 호출되는 함수입니다. (기기 성능에 따라 초당 60~120회 이상 호출)

5. 소스 파일 (Item.cpp)과 Super 구조¶

 "Item.h"

AItem::AItem()
{
    PrimaryActorTick.bCanEverTick = true;
}

void AItem::BeginPlay()
{
    Super::BeginPlay();
}

void AItem::Tick(float DeltaTime)
{
    Super::Tick(DeltaTime);
}

  • Super::BeginPlay(), Super::Tick(DeltaTime) 처럼 Super::를 사용하는 것은 부모 클래스(Actor)가 내부적으로 수행해야 하는 기본 동작을 먼저 실행하도록 보장하기 위함이므로 반드시 유지해야 합니다.

  • 성능 최적화 팁: 생성자에 있는 PrimaryActorTick.bCanEverTick = true;는 이 액터가 매 프레임 Tick 함수를 실행하도록 설정하는 것입니다. 만약 매 프레임 연산이 필요 없는 액터라면 이 값을 false로 바꾸어 성능을 아낄 수 있습니다.

6. 코드 리팩토링 (깔끔하게 정리하기)¶

강의자는 화면을 더 깔끔하고 가독성 좋게 만들기 위해 아래와 같이 코드를 정리했습니다.

  • 위아래로 나뉘어 있던 두 개의 public: 섹션을 하나로 합치고, Tick 함수를 상단의 public: 영역으로 이동시켰습니다.
  • 이미 내용을 알고 있는 기본 생성 주석들(예: "called when the game starts" 등)을 모두 삭제하여 클래스 구조를 더 콤팩트하게 만들었습니다.

Blueprint Creation¶

C++과 블루프린트의 관계 및 생성¶

  • 원칙: 우리는 보통 생 C++ 클래스 자체를 레벨에 직접 배치하지 않습니다. C++ 클래스를 부모(Parent)로 삼고, 이를 상속받은 블루프린트(Child)를 만들어 에디터에서 시각적 기능(메시, 컴포넌트 등)을 추가해 사용합니다.

  • 생성 방법:

    1. 콘텐츠 브라우저에 Blueprints/Items 폴더 생성.
    2. C++ 클래스(Item)를 우클릭하여 'Item 기반 블루프린트 클래스 생성' 을 선택하거나, 블루프린트 생성 창에서 All Classes를 펼쳐 Item을 검색해 상속받습니다.

  • 네이밍 컨벤션: C++ 클래스와 구분하기 위해 블루프린트 에셋 이름 앞에 BP_ 접두사(Prefix)를 붙입니다. (예: BP_Item)

블루프린트 에디터 구조와 주요 탭¶

BP_Item을 더블 클릭하면 블루프린트 에디터가 열립니다.

  • Viewport(뷰포트): 액터의 컴포넌트 구조와 로컬 좌표계(X, Y, Z축 기즈모)를 시각적으로 확인하는 곳입니다. 모든 액터는 기본적으로 Default Scene Root 컴포넌트를 가집니다.

  • Construction Script(구조 스크립트): 게임이 시작되기 전, 에디터에서 프로퍼티가 변경되거나 액터가 움직일 때마다 즉시 실행되는 스크립트입니다. 사전 세팅에 유용합니다.

  • Event Graph(이벤트 그래프): 게임 플레이 중(런타임) 실행될 로직을 노드로 짜는 곳입니다. 기본적으로 BeginPlay, ActorBeginOverlap, Tick 이벤트 노드가 제공됩니다.

  • 컴파일(Compile): 블루프린트 노드를 수정하면 C++ 코드처럼 반드시 상단의 '컴파일' 버튼을 눌러주어야 변경 사항이 반영됩니다.

디버깅 기초: Print String과 액터의 중요도(Relevance)¶

  • Print String 노드: BeginPlay 이벤트에 연결하여 화면에 "Begin Play"라는 텍스트를 출력할 수 있습니다. 텍스트 색상이나 출력 시간(기본 2초에서 테스트를 위해 30초로 변경)을 상세 옵션에서 조절 가능합니다.

  • 오픈 월드에서의 액터 중요도(Relevance):

    • 오픈 월드 맵에서 플레이어(카메라) 위치가 BP_Item과 너무 멀리 떨어져 있으면, 엔진은 성능 최적화를 위해 해당 액터가 존재하지 않는 것처럼 취급하여 BeginPlay를 실행하지 않습니다. 메시지를 확인하려면 아웃라이너에서 액터를 선택하고 F 키를 눌러 가까이 이동(포커스)한 뒤 플레이해야 합니다.

C++ 디버깅의 핵심: UE_LOG 매크로¶

블루프린트의 Print String처럼, C++에서는 UE_LOG 매크로를 사용해 에디터의 출력 로그(Output Log) 창에 원하는 메시지를 출력합니다.

UE_LOG 작성 코드 분석¶

void AItem::BeginPlay()
{
    Super::BeginPlay();

    UE_LOG(LogTemp, Warning, TEXT("Begin Play Called"));
}

  1. LogTemp (로그 카테고리): 디버깅용으로 흔히 쓰는 임시 로그 카테고리입니다.

  2. Warning (상세 수준/Verbosity): 로그의 중요도입니다. Warning으로 설정하면 출력 로그 창에 노란색으로 표시되어 눈에 잘 띕니다.

  3. TEXT("문자열") 매크로: 언리얼 엔진은 문자열 인코딩으로 유니코드(Unicode)를 사용합니다. 일반 문자열 리터럴을 언리얼 전용 문자열(FString 계열)에 맞는 유니코드 형식으로 안전하게 변환하기 위해 반드시 모든 문자열을 TEXT() 매크로로 감싸야 합니다. (언리얼 코딩 표준 권장 사항)

데이터 적용의 우선순위 (어떤 값이 최종 반영되는가?)¶

변수의 '값'이 최종적으로 무엇으로 결정되는지에 대한 우선순위(오버라이드 권한)는 다음과 같습니다. 뒤로 갈수록 앞에 설정한 값을 덮어씁니다.

C++ 생성자 초기값 -> 에디터 디테일 패널 수정값 -> 블루프린트 Construction Script 연산값

  1. 최하위 (C++ 생성자): 가장 먼저 기본값을 설정하지만, 에디터나 블루프린트에서 언제든 덮어쓸 수 있습니다.

  2. 중간 (에디터 디테일 패널): 개발자가 에디터 창에서 수동으로 입력한 값입니다. C++ 생성자에 적힌 값을 무시하고 덮어씁니다.

  3. 최상위 (Construction Script): 에디터에서 값을 어떻게 바꿨든 간에, 컴포넌트가 배치되거나 움직일 때 Construction Script의 노드가 실행되면서 최종적으로 값을 강제 지정해 버립니다.

⚠️ 주의: 만약 Construction Script 내부에서 특정 변수의 값을 지정하는 노드(Set 변수)를 짜놓았다면, 에디터 디테일 패널에서 아무리 값을 변경해도 컴파일이나 에디터 배치 직후 Construction Script에 의해 값이 다시 초기화됩니다.

액터 생성 및 에디터 배치 시 실행 순서 (Life Cycle)¶

액터가 에디터에 배치되거나, 게임이 시작될 때 내부적으로 함수가 호출되는 정확한 메커니즘과 순서는 다음과 같습니다.

C++ 생성자 (Constructor)
        |
데이터 직렬화 및 로드 (Data Serialization)
        |
PostInitProperties() 호출 (C++ 내부 함수)
        |
블루프린트 Construction Script 실행
        |

① C++ 생성자 (Constructor) 호출¶

  • 시점: 에디터가 켜질 때 또는 액터가 월드에 배치되려고 메모리에 올라가는 순간.
  • 역할: UPROPERTY 변수들의 완전한 초기 기본값을 세팅합니다.

② 데이터 직렬화 및 로드 (Data Serialization)¶

  • 시점: 생성자 호출 직후.
  • 역할: 에디터 디테일 패널에서 개발자가 수정한 값(저장된 데이터)이 C++ 생성자가 만든 변수 위에 덮어씌워집니다.

③ PostInitProperties() 호출 (C++ 내부 함수)¶

  • 시점: 생성자 기본값에 에디터 수정 사항까지 모두 반영된 직후.
  • 역할: 에디터에서 수정된 최종 프로퍼티 값을 기반으로, C++ 레벨에서 추가적인 초기화 작업을 하고 싶을 때 호출되는 함수입니다.

④ 블루프린트 Construction Script 실행¶

  • 시점: 에디터에서 액터를 드래그하여 움직일 때(매 순간), 혹은 컴파일 버튼을 누를 때, 레벨에 처음 배치될 때 실행됩니다.
  • 역할: 앞서 2번 단계까지 완료되어 들어온 변수 값들을 가져와 가공하거나, 새로운 값을 대입(Set)합니다. 블루프린트 기반의 동적 레벨 제작(예: 변수 값에 따라 스태틱 메시 모양 바꾸기)이 주로 여기서 일어납니다.

⑤ OnConstruction() 호출 (C++ 내부 함수)¶

  • 시점: 블루프린트의 Construction Script가 실행 완료된 바로 직후.
  • 역할: 블루프린트 Construction Script의 연산 결과까지 포함된 최종 데이터를 가지고 C++에서 마지막으로 컴포넌트를 정렬하거나 세팅할 때 사용합니다.
