C++ Refresher¶

C++ 객체 지향 프로그래밍(OOP) 핵심 개념 복습¶

1) 상속(Inheritance)과 다형성(Polymorphism)¶

  • 상속 계층 구조 (Inheritance Chain): 부모 클래스의 변수와 함수를 자식 클래스가 그대로 물려받으며, 자식의 자식으로 무한히 확장될 수 있습니다. (다중 상속도 가능)

  • 가상 함수와 오버라이드: 부모가 함수를 virtual로 선언하면, 자식은 이를 오버라이드(Override)하여 자신만의 기능을 재정의할 수 있습니다.

  • 다형성(Polymorphism): 부모 클래스 타입의 포인터 변수(Parent*)는 자신을 상속받은 자식 객체들을 모두 가리킬 수 있습니다. 이때 가상 함수를 호출하면, 포인터 타입이 아닌 '실제 가리키고 있는 객체'의 오버라이드된 함수가 호출됩니다.

2) 다운캐스팅(Downcasting)과 동적 캐스팅(Dynamic Cast)¶

  • 문제 상황: 부모 포인터로 자식 객체를 가리키고 있을 때, 자식 클래스에만 존재하는 고유 함수는 바로 호출할 수 없습니다. 컴파일러는 부모의 인터페이스만 보기 때문입니다.

  • 해결책 (다운캐스팅): 부모 포인터 타입을 자식 포인터 타입으로 변환해야 합니다.

  • 동적 캐스팅 (Dynamic Cast): 런타임(게임 실행 중)에 실제 객체의 타입을 검사하여 안전하게 캐스팅합니다.

    • 성공 시: 자식 타입 포인터를 반환하여 고유 함수 호출 가능.
    • 실패 시: 객체 타입이 일치하지 않으면 nullptr(Null)을 반환하므로, 캐스팅 후에는 반드시 Null 체크를 해야 안전합니다.

관계의 종류: "Is-A" vs "Has-A"¶

  • Is-A 관계 (상속): "~은 ~이다."

    • 예: 자식은 부모이다(O), 캐릭터는 액터이다(O). 하지만 부모는 자식이 아니다(X).

  • Has-A 관계 (포함/소유): "~은 ~을 가지고 있다." 클래스 내부에 멤버 변수로 다른 클래스나 컴퍼넌트를 소유하는 관계입니다.

    • 예: 캐릭터는 무기를 가지고 있다, 액터는 컴포넌트를 가지고 있다.

언리얼 엔진(Unreal Engine) 클래스 Hierarchy¶

  • 언리얼 엔진의 거의 모든 클래스는 거대한 상속 체인으로 묶여 있으며, 이름 앞에 고유한 접두사(Prefix)가 붙습니다.

핵심 클래스 4단계 상속 구조¶

  • UObject (접두사: U)

    • 언리얼에서 가장 기본이 되는 최상위 클래스입니다.
    • 데이터 저장, 가비지 컬렉션(메모리 관리), 리플렉션 등의 핵심 기능을 제공하지만, 단독으로는 게임 월드에 배치될 수 없습니다.

  • AActor (접두사: A)

    • UObject를 상속받았으며, 게임 레벨(월드)에 배치될 수 있는 모든 오브젝트의 기본형입니다.
    • 눈에 보이는 메시(Mesh)나 다양한 기능성 컴포넌트들을 가질 수 있습니다.

  • APawn (접두사: A)

    • AActor를 상속받았으며, 플레이어나 AI가 ‘빙의(Possess)’하여 조종할 수 있는 액터입니다.
    • 사용자의 키보드/마우스 입력이나 AI 컨트롤러의 명령을 받아 움직일 수 있는 물리적 체형을 가집니다.

  • ACharacter (접두사: A)

    • APawn을 상속받았으며, 인간형(이족보행) 캐릭터에 특화된 정교한 클래스입니다.
    • 걷기, 달리기, 점프, 낙하 등 복잡한 이동 물리 계산이 이미 구현된 ‘Character Movement Component’를 기본적으로 내장하고 있습니다.

폰(Pawn) vs 캐릭터(Character) 선택 기준¶

  • 단순히 입력받아 이리저리 움직이는 기본적인 크리처나 비행기 등은 Pawn으로도 충분하지만, 뛰어난 이족보행 물리 및 네트워킹 이동 동기화가 필요한 캐릭터를 만들 때는 Character를 선택하는 것이 훨씬 유리합니다.

언리얼 월드의 Has-A (소유) 중첩 구조¶

  • 언리얼 엔진은 거대한 내부적 포함 관계를 가집니다. 이를 서브 오브젝트(Subobject) 관계라고 부릅니다.

  Package -> World -> Level -> Actor -> Component

* Package (패키지): 최외곽 에셋/맵 데이터 단위
* World (월드): 게임이 구동되는 전체 세계
* Level (레벨): 월드 내에 존재하는 하나의 스테이지/맵 공간
* Actor (액터): 레벨에 배치되어 소유되는 객체들
* Component (컴포넌트): 액터의 기능을 확장하기 위해 액터가 소유하는 부속품들 (ex: 스태틱 메시 컴포넌트, 오디오 컴포넌트 등)

Reflection and Garbage Collection¶

리플렉션(Reflection) 시스템이란?¶

💡 쉽게 말해: 프로그램이 실행 중(Runtime)에 "나 자신을 스스로 들여다보고 분석하는 능력" 입니다. 내 안에 어떤 클래스가 있고, 그 클래스 안에 어떤 변수와 함수가 있는지 스스로 정보를 수집(Harvest)하는 시스템입니다.

표준 C++에는 없고, 언리얼에는 있는 이유¶

  • 기본 C++은 컴파일이 끝나면 클래스나 변수의 '이름' 같은 정보가 모두 사라지고 바이너리 코드만 남습니다. 하지만 언리얼 엔진은 아래의 강력한 기능들을 구현하기 위해 자체적인 리플렉션 시스템을 구축했습니다.

  • 에디터 연동: C++로 만든 변수를 언리얼 에디터(디테일 패널)에 노출시켜 기획자가 값을 수정할 수 있게 합니다.

  • 블루프린트 연동: C++ 함수와 변수를 블루프린트 노드에서 불러다 쓸 수 있게 연결해 줍니다.
  • 가비지 컬렉션: 어떤 객체가 현재 게임에서 사용 중인지 추적하여 자동으로 메모리를 관리합니다.

가비지 컬렉션(Garbage Collection)과 메모리 관리¶

  • 표준 C++에서는 new 키워드로 동적 할당한 객체를 다 쓰고 나면 반드시 delete로 메모리를 해제해야 합니다. 깜빡하면 메모리가 새어 나가는 메모리 누수(Memory Leak)가 발생.

언리얼의 자동 메모리 삭제 원리¶

언리얼 엔진의 리플렉션 시스템에 등록된 객체(UObject 계열)들은 가비지 컬렉터가 늘 지켜보고 있습니다.
* 포인터 변수가 객체를 가리킬 때마다 참조 횟수(Reference Count)를 기록합니다.
* 만약 해당 객체를 가리키는 변수가 스코프를 벗어나거나 사라져서 참조 횟수가 0이 되면, 게임에 더 이상 필요 없는 '쓰레기(Garbage)'로 판단합니다.
* 가비지 컬렉터가 런타임에 이 객체를 자동으로 delete하여 메모리를 해제해 줍니다. 개발자가 직접 메모리를 지울 필요가 없어 안전합니다.

리플렉션을 발동시키는 마법의 주문: 매크로(Macro)¶

C++ 코드에게 리플렉션 시스템과 가비지 컬렉션에 참여하라고 알려주려면 언리얼 고유의 매크로를 코드에 붙여야 합니다.

※ 매크로(Macro)란? 특정 코드 뭉치에 이름을 붙여둔 것입니다. 컴파일하기 직전(전처리 단계)에 매크로 이름이 실제 복잡한 코드 코드로 촥 펼쳐지게 됩니다.

핵심 매크로 3총사¶

  • UCLASS(): 클래스 선언문 바로 위에 붙입니다. 이 클래스가 언리얼 시스템의 관리를 받겠다는 선언입니다.
  • UPROPERTY(): 멤버 변수 위에 붙입니다. 에디터 노출 및 가비지 컬렉션 대상이 됩니다.
  • UFUNCTION(): 멤버 함수 위에 붙입니다. 블루프린트에서 호출할 수 있게 해줍니다.

빌드의 숨은 일꾼: UHT (Unreal Header Tool)¶

우리가 C++ 코드를 작성하고 컴파일(빌드)을 누르면, 가장 먼저 UHT(Unreal Header Tool)라는 프로그램이 출동합니다.

 "MyCharacter.generated.h" // 2번: UHT가 만든 파일을 포함시킴

UCLASS() // 1번: 매크로 발견!
class AMYCHARACTER_API AMyCharacter : public ACharacter
{
    GENERATED_BODY() // 3번: UHT가 생성한 코드가 여기에 주입됨

    UPROPERTY(EditAnywhere)
    float Health;
};

UHT의 작동 순서¶

  • 코드 분석: UHT가 헤더 파일(.h)을 훑으며 UCLASS(), UPROPERTY() 같은 매크로를 찾아냅니다.

  • 부가 코드 자동 생성: 매크로를 발견하면, 언리얼 엔진이 이 클래스를 이해할 수 있도록 돕는 복잡한 C++ 코드를 배후에서 자동으로 만들어 냅니다.

  • "*.generated.h"와 GENERATED_BODY(): UHT가 자동으로 만든 코드는 파일 상단의 파일명.generated.h에 저장되며, 클래스 내부의 GENERATED_BODY() 위치에 쏙 들어가 결합됩니다. (언리얼 헤더 파일 상단에 항상 이 include문이 있어야 하고, 클래스 내부에 이 매크로가 필수인 이유입니다.)

지정자(Specifiers) 활용¶

매크로 뒤의 괄호 안에 UPROPERTY(EditAnywhere, BlueprintReadWrite)처럼 지정자(Specifier)를 수 있습니다. 일종의 '옵션 매개변수' 역할을 하여, 이 변수를 에디터에서 수정만 가능하게 할지, 블루프린트에서 읽고 쓰기까지 가능하게 할지 세부적으로 제어할 수 있습니다.
